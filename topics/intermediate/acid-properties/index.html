ACID Properties

Before this topic: you should be comfortable writing basic SELECT, INSERT, UPDATE and DELETE queries, and unterstand what a transaction is - that a BEGIN / COMMIT block is a unit of work that either fully succeeds or fully rolls back.

Every transaction in SQL Server must satisfy four guarantees. Together they are known as ACID - a shorthand for Atomicity, Consistency, Isolation and Durability. These are not features you turn on. They are promises the database engine makes to you about every transaction, all the time.

To make each guarantee concrete, the same scenario runs through all four: a bank transfer moves $200 from a savings account to a current account. That is two UPDATE statements inside one transaction. Simple - but each guarantee is protecting you from a different way it could wrong.

Atomicity

The guarantee

Every statement in a transaction either all succeeds or all fails together. There is no partial outcome.

The transfer requires two steps: debit savings, credit current. Atomicity means both happen, or neither happens. If the server crashes between the debit and the credit, SQL Server rolls back the debit automatically on restart - the account is exactly as it was before the transaction started. The $200 cannot disappear into a gap between two statements.

The mechanism is the transaction log. Before any data page is changed, SQL Server writes the intended change to the log. If revocery is needed, the engine replays or undoes entries from the log to restore a consistent state.

Consistency

The guarantee

A transaction can only bring the database from one valid state to another. It cannot leave the database in a state that violates defined rules and constraints.

The bank enforces a rule: no account balance may go below $0. That rule is defined as a CHECK constraint on the Account table. If the savings balance is only $150 and the transfer requests $200, the transaction violates the constraint. SQL Server rejects the entire transaction and rolls it back. The database stays valid.

Consistency is the only ACID property that is partly your responsibility. SQL Server enforces constraints you define - foreign keys, CHECK constraints, UNIQUE indexes, NOT NULL. But if a business rule is encoded as a constraint, the database cannot enforce it. Consistency is a partnership between the engine and the schema designer.

Isolation

The guarantee

Concurrent transactions behave as if they ran one after another. Intermediate states are not visible to other transactions.

While the transfer is mid-flight - savings debited, current not yet credited - a second transaction reads both balances. Without isolation, it sees $200 missing from savings but not yet present in current. It makes a decision based on a state that never permanently existed. With isolation, the intermediate state is hidden until the transfer commits.

Isolation is the one ACID property with adjustable trade-offs. Perfect isolation is expensive - it requires locking, which slows concurrent work. SQL Server lets you dial this trade-off using isolation levels. That is why isolation levels have their own topic page - the I in ACID is the letter with settings.

Durability

The guarantee

Once a transaction commits, its changes survive permanently - even if the server loses power one second later.

The transfer commits at 14:32:07. At 14:32:08 the server loses power. When it restarts, the transfer is still there. SQL Server guarantees this by writing committed changes to the transaction log on disk before returning success to the application. The data pages in memory may not yet be flushed to disk - but the log is and the log is enough to reconstruct the committed state on revocery.

This is sometimes called write-ahead (WAL): the log is always written before the data page is written. It is the foundation of both Durability and atomicity.

How the four properties connect

Atomicity and Durability are enforced by the same mechanism - the transaction log. Atomicity uses the log to undo incomplete work; Durability uses it to survive crashes and replay committed work. Consistency relies on your schema design - the engine enforces constraints you define. Isolation is the only property with a dial: the five isolation levels in SQL Server are five different answers to the question: "how much isolation are you willing to pay for?"

The I in ACID has settings. Atomicity, Consistency and Durability are binary - the engine either provides them or it does not.
