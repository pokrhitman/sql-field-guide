Transactions

Before this topic: you should be comfortable writing basic SELECT, INSERT, UPDATE and DELETE queries. It also helps to have read ACID Properties first - this stage explains the mechanics behind the guarantees that ACID Properties introduces.

A transaction is a unit of work: one or more SQL statements that the database treats as a single, indivisible operation. Either every satement in the transaction succceeds and the changes are permanent, or something goes wrong and every change is undone as if the transaction never ran. There is no middle ground.

Four keywords control the lifecycle of a transaction in SQL Server. Before looking at the syntax, it helps to have a mental mode that makes all four obvious.

The undo stack

Think of an active transaction as a stack of changes. Every INSERT, UPDATE or DELETE you execute inside the transaction is pushed onto the stack. The stack is private to your transaction - no other session can see it. SQL Server records everything needed to reverse each change.

When the transaction ends, one of two things happens to the stack:

COMMIT --discard the stack

The changes are written permanently to the database. The stack is no longer needed and is discarded. There is nothing left to undo.

ROLLBACK --pop the entire stack

Every change on the stack is reversed, in reverse order, back to the state before the transaction started. the database is restored exactly as it was.

This mental model also explains why uncomitted changes are invisible to other transactions: the stack is private. Other sessions only see the data after a COMMIT makes the changes public.

The four keywords

BEGIN TRANSACTION

Opens the transaction and starts the stack. Every statement from this point forwards is part of the transaction until it is ended with COMMIT and ROLLBACK.

COMMIT

Ends the transaction successfully. All changes are made permanent and visible to other sessions. The stack is discarded.

ROLLBACK

Ends the transaction by undoing all changes. The database returns to the state it was before BEGIN TRANSACTION. Typically used in a CATCH block when an error is detected.

SAVE TRANSACTION (savepoint)

Places a named marker in the stack without ending the transaction. A subsequent ROLLBACK TO that name pops the stack back to the marker - undoing only the changes made after it - while leaving the transaction open. Changes made before the savepoint are preserved.

Why savepoints exist: a real example

An e-commerce platform processes an incoming order in three steps, all inside one transaction:

Step 1 - Insert a new row into the Orders table.
Step 2 - Deduct the oredered quantity from the Inventory table.
Step 3 - Insert a row into the FullfilmentLog table recording the outcome.

The business rule is: if Step 2 fails because the item is out of stock, do not cancel the order - keep the order record and log a backorder status instead. Only the stock deduction should be undone.

Without a savepoint, any error would trigger a full ROLLBACK, losing the order record along with the failed stock deduction. With a savepoint placed after Step 1, the transaction can roll back to that marker on a Step 2 failure, preserving the order record, and then continue with Step 3 using the backorder outcome.

Syntax patterns

Basic transaction

BEGIN TRANSACTION;

The simplest pattern: open a transaction, do the work, commit. If anything goes wrong before COMMIT, call ROLLBACK to undo everything.

Transaction with error handling

BEGIN TRY ... END TRY BEGIN CATCH ... END CATCH;

Wrap the transaction in a TRY/CATCH block. If any statement raises an error, execution jumps to the CATCH block. Check XACT_STATE() before rolling back - after some errors SQL Server has already rolled back automatically.

Savepoint with a partial rollback

SAVE TRANSACTION savepoint_name;

Place a named marker in the transaction. Use ROLLBACK TRANSACTION savepoint_name to undo only the work done after a marker, leaving the transaction open and earlier changes intact.

Savepoint syntax across dialetcs

Standard SQL, MySQL and PostgreSQL use a different savepoint syntax to T-SQL. If your coursework or prior experience used a different database, the keywords will look unfamiliar here.

Action Standard SQL / MySQL / PostgreSQL T-SQL (SQL Server)
Create a savepoint SAVEPOINT name SAVE TRANSACTION name
Roll back to savepoint ROLLBACK TO SAVEPOINT name ROLLBACK TRANSACTION name
Release a savepoint RELEASE SAVEPOINT name Not supported - saveppoints are discarded automatically on COMMIT or full ROLLBACK
T-SQL trap: ROLLBACK TRANSACTION with and without a name behave differently.
ROLLBACK TRANSACTION - no argument - undoes everything back to the BEGIN and ends the transaction.
ROLLBACK TRANSACTION name - with a savepoint name - undoes only the work done after that savepoint and leaves the transaction open.

The keyword is identical. The presence or absence of the savepoint name is the only difference. This is a common source of bugs when moving from other databases to SQL Server, or when reading T-SQL code for the first time.

T-SQL syntax reference

-- Basic transaction BEGIN TRANSACTION; INSERT INTO Orders (CustomerID, Total) VALUES (101, 249.99); UPDATE Inventory SET Stock = Stock - 1, WHERE ProductID = 42; COMMIT; -- Transaction with TRY/ CATCH and XACT_STATE() check BEGIN TRY BEGIN TRANSACTION; INSERT INTO Orders (CustomerID, Total) VALUES (101, 299.99); UPDATE Inventory SET Stock = Stock - 1 WHERE ProdcutID = 42; COMMIT; END TRY BEGIN CATCH IF XACT_STATE() <> 0 ROLLBACK TRANSACTION; THROW; END CATCH; -- Savepoint with partial rollback BEGIN TRANSACTION; INSERT INTO Orders (CustomerID, Total) VALUES (101, 249.99; SAVE TRANSACTION after_order; -- marker placed here BEGIN TRY UPDATE Inventory SET Stock = Stock - 1WHERE ProductID = 42; END TRY BEGIN CATCH ROLLBACK TRANSACTION after_order; -- undo only from marker INSERT INTO FullfilmentLog (OrderID, Status) VALUES (101, 'Backorder'); END CATCH COMMIT;

XACT_STATE() why it matters in CATCH blocks

Returns 1 if a transaction is open and commitable, -1 if it is open but uncommittable (must be rolled back), or 0 if there is no active transaction. Always check before calling ROLLBACK in a CATCH block - calling ROLLBACK when no transaction is active raises a second error. Key terms

Term Meaning
BEGIN TRANSACTION Opens a transaction. All subsequent statements are part of this unit of work.
COMMIT Makes all changes permanent and visible to other sessions. Ends the transaction.
ROLLBACK Undoes all changes back to the BEGIN (or to a named savepoint). Ends the transaction.
SAVE TRANSACTION Places a named marker inside an open transaction. Does not end the transaction.
ROLLBACK TO savepoint Undoes changes made after the named savepoint. Transaction remains open.
XACT_STATE() Returns the state of the current transaction: 1 (active), -1 (uncommittable), 0 (none).
