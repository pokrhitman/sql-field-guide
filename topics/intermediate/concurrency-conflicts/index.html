Concurrency Conflicts

Before this topic you should understand what a transaction is (BEGIN / COMMIT / ROLLBACK) and have a basic familiarity with the isolation levels — specifically that transactions hold locks on the data they read or modify. Isolation Levels →

Choosing the right isolation level handles most concurrency problems in theory. In practice, two situations still break through: deadlocks, where the engine has to forcibly kill a transaction, and commit failures under Snapshot isolation, where the engine detects a conflict too late to prevent it. Both require your application — not just your SQL settings - to respond correctly.

What is a deadlock?

A deadlock happens when two transactions are each waiting for the other to release a lock and neither can continue. Each one holds something the other needs. Left alone, they would wait forever.

SQL Server's deadlock monitor (the lock monitor thread) wakes up every five seconds to scan for these cycles. When it finds one, it picks a "victim" — usually the transaction that would cost less to roll back — and terminates it with error 1205. The victim's transaction is rolled back automatically. The surviving transaction then gets its lock and continues.

How the victim is chosen

SQL Server picks the transaction with the lowest deadlock priority. If both are equal (the default), it rolls back whichever transaction is cheaper to undo — typically the one that has done less work so far. You can influence this with SET DEADLOCK_PRIORITY LOW | NORMAL | HIGH, which is useful when one transaction is much more expensive to re-run than the other.

Why deadlocks happen — and how to reduce them

Deadlocks almost always come from transactions accessing the same tables in different orders. if Transaction A always locks Orders then Customers, and Transaction B always locks Customers then Orders, a deadlock is only a matter of timing. The fix is consistent lock ordering: make both transactions access tables in the same sequence.

Short transactions also help. The longer a transaction holds locks, the larger the window in which another transaction can collide with it. Keeping transactions focused — do the work, commit quickly — reduces the opportunity for overlap.

What is a commit failure under Snapshot?

Snapshot isolation avoids deadlocks by giving each transaction its own frozen copy of the data (a "snapshot" taken at the moment the transaction starts). Because readers never block writers and writers never block readers, deadlocks largely disappear. But there is a trade-off: SQL Server cannot prevent two transactions from both reading the same data and then both trying to update it. Instead it detects the conflict at commit time and throws error 3960, rolling back the second writer.

This is called an update conflict. It is not a bug — it is Snapshot doing its job. But your application must expect it and know what to do next.

Retry logic: the correct response to both errors

Both error 1205 (deadlock victim) and error 3960 (Snapshot update conflict) mean: your transaction was rolled back, try again. The data is in a consistent state. The work simply needs to be re-attempted

The pattern is always the same: catch the specific error number, wait a short time, and retry — up to a maximum number of attempts. The wait before retrying is important: if two transactions conflicted once at exactly the same moment, retrying immediately at the same moment will conflict again. A small random delay breaks the synchronisation.

Reducing deadlocks

Consistent lock ordering

If multiple transactions touch the same tables, always access them in the same order. This prevents the circular wait that causes deadlocks.

Short transactions

Commit as quickly as possible. Every second a transaction holds a lock is a second another transaction could collide with it.

Deadlock priority

SET DEADLOCK_PRIORITY LOW | NORMAL | HIGH;

Tells SQL Server which transaction to sacrifice first. Set LOW on transactions that are cheap to retry; set HIGH on expensive long-running ones.

T-SQL retry pattern

Apply this wrapper around any transaction that might be chosen as a deadlock victim or hit a Snapshot update conflict. Errors 1205 and 3960 both mean "rolled back — safe to retry."

--Retry wrapper for deadlock (1205) and Snapshot conflict (3960) -- Copy this pattern and replace the inner block with your transaction. DECLARE @attempt INT = 1; DECLARE @maxTries INT = 3; DECLARE @success INT = 0; WHILE @attempt <= @maxTries AND @success= 0 BEGIN BEGIN TRY BEGIN TRANSACTION; --Your work goes here UPDATE Orders SET Status = 'Processed' WHERE OrderID = 42; UPDATE Customers SET LastOrder = GETDATE() WHERE CustomerID = 7; COMMIT TRANSACTION; SET @success = 1; -- Reached here = no error, exit the loop END TRY BEGIN CATCH IF XACT_STATE <> 0 ROLLBACK TRANSACTION -- Clean up before retrying IF ERROR_NUMBER() IN (1205, 3960) AND @attempt < @maxTries BEGIN -- Retryable error: wait briefly, then try again -- WAITFOR adds a small delay so a retry does not hit the same -- collision at exactly the same millisecond. WAITFOR DELAY '00:00:00.200' -- 200ms SET @attempt = @attempt + 1; END ELSE BEGIN -- Different error, or retries exhausted: re-raise THROW END END CATCH END

Why check XACT_STATE() before rolling back?

After error 1205, SQL Server has already rolled back your transaction automatically. Calling ROLLBACK on something that no longer exists causes a second error. XACT_STATE() returns 0 if there is no active transaction, -1 if it is uncommittable (must be rolled back), or 1 if it is still valid. Checking first makes the CATCH block safe in all three cases.

MySQL / PostgreSQL note

MySQL uses error code 1213 for deadlock victim. PostgreSQL raises SQLSTATE 40P01. The retry logic pattern is identical - only the error number to catch changes.
