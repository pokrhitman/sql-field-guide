Locks

Prerquisite: Transaction. Locks are the mechanism the database engine uses to enforce everything a transaction promises - this page explains how that promise is actually kept.

Mental model: claim tickets

When a transaction reads or writes a row, it takes out a "claim ticket" on that row. Most ticket types let other transactions take a number and queue behind it - they'll get their turn once the ticket is released at COMMIT or ROLLBACK. But some ticket types mean nobody else can even queue: they have to wait until the ticket holder is completely done. This is the difference between a shared lock (read ticket - others can queue for reads too) and an exclusive lock (write ticket - everyone else waits).

Four problems, four answers

Locking exists to solve four concrete problems. Each one maps directly onto a piece of SQL Server's lock vocabulary you'll meet in the Reference tab.

1. Lost updates → exclusive locks (X)

Two transactions both read a value, both calculate a new value based on it and both write back - the second write silently overwrites the first and one update is lost. An exclusive lock on the row being modified prevents a second writer from touching it until the first is finished.

2. Dirty reads → shared locks (S) blocking on X

Two transactions a row that another transaction has changed but not yet committed. If that change is later rolled back, the first transaction acted on data that never really existed. A shared lock request has to wait if an exclusive lock is already held on that row - so readers don't see half-finished writes.

3. Deadlocks → the deadlock detector

Two transactions each hold a lock the other one needs and each is waiting for the other to release it. Neither ever will. SQL Server periodically checks for these cycles and picks one transaction as the deadlock victim - it gets rolled back automatically with error 1205, freeing its locks so the other transaction can continue.

4. Lock escalation → granularity (row/ page/ table)

Locks aren't free to track - each one is a small amount of memory. If a single statement is about to acquire a very large number of row-level locks, SQL Sever escalates to a smaller number of coarser locks (page or table level) to keep overhead down. The tradeoff: a table-level lock blocks everyone from writing to that table, not just the rows actually being changed.

Scenario: two work-order stations, one inventory table

A manufacturing plant tracks component stock in an inventory table, keyed by PartID. Two assembly stations each run a work order that consumes two different parts. Station A's work order needs Part 501 first, then Part 620. Station B's work order needs Part 620 first, then Part 501 - the same two parts, in the opposite order. Both stations start their Transactions at almost the same moment.

Each station's first UPDATE succeeds immediately and takes an exclusive lock on that part's row. But when each station tries to update its second part, that row is already exclusively locked by the other station - and neither will release its first lock until its transaction commits. This is exactly the deadlock cycle from problem 3. Walk through it step by step in the Interactive tab.

Lock modes

ModeAbbreviationUsed forBlocks
SharedS Reading a row (default isolation level) Exclusive locks on the same row
ExclusiveX INSERT, UPDATE, DELETE Shared and exclusive locks on the same row
UpdateU Rows a statement might update, before it commits to it Other U and X locks (but not S) - avoids a common deadlock pattern between two readers that both intend to write
IntentIS / IX Signals at table/page level that a more granular lock is held below lets SQL Server check for conflicts at a higher level without scanning every row

Lock granularity

LevelScopeNotes
RowA single row Finest granularity, lowest blocking impact, highest per-lock overhead
PageAn 8KB data page (multiple rows) Used when row-level would mean tracking too many locks
TableThe entire table Coarsest - blocks all writers (and sometimes readers) to the table until released
Lock escalation: SQL Server can automatically escalate row or page locks held by a single statement to a table lock once a treshold (around 5,000 locks) is reached. This is automatic and not directly configurable per query - it's a reason a single large batch UPDATE or DELETE can unexpectedly block unrelated queries on the same table, even ones touching completely different rows.
Seeing locks in SSMS

Current locks

SELECT * FROM sys.dm_tran_locks;

Lists every lock currently held or being waited on, including the resource type (RID, PAGE, OBJECT), the lock mode and the session that holds it. Useful for confirming what a blocked query is actually waiting behind.

Who's blocking whom

SELECT session_id, blocking_session_id, wait_type, wait_time FROM sys.dm_exec_requests WHERE blocking_session_id <> 0;

Any row here means one session is waiting on a lock held by another. A non-zero blocking_session_id is the fastest way to confirm "is something blocked right now" before digging into sys.dm_tran_locks for detail.

Deadlock error

Error 1205

Transaction (Process ID 56) was deadlocked on Lock resources with another process and has been chosen as the deadlock victim. Rerun transaction.

The victim's transaction is automatically rolled back - its locks released so the other transaction can proceed. The error message itself even tells you what to do: catch it and retry. See the Concurrency Conflicts page for the retry pattern using XACT_STATE().

Key terms

TermMeaning
Blocking A transaction waiting for a lock by another. Normal and expected - resolves once the holder commits or rolls back.
Deadlock A cycle of blocking with no possible resolution. SQL Server detects and breaks it by rolling back one side.
Lock Escalation Automatic promotion form row/page locks to a table lock once a treshold is crossed.
Granularity The size is the resource a lock covers - row, page or table.
