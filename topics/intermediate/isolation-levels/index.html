Isolation Levels

Before this topic: you should be comfortable writing basic SELECT queries and understand what a transaction is - that a BEGIN / COMMIT block is a unit of work that either fully succeeds or fully rolls back.

When multiple users query or change data at the same time, things can go wrong in predictable ways. Isolation levels are SQL Server's setting for controlling how much one transaction can be disturbed by others running at the same time.

The higher the isolation, the safer the data - but the more other transactions have to wait. The lower the isolation, the faster everything runs - but the more risk of reading inaccurate data.

The four problems isolation levels protect against

Dirty read

You read data that another transaction changed but has not committed yet. If tha transaction rolls back, you read something that never really existed.

Non-repeatable read

You read a row, then read it again in the same transaction and get a different value - because someone else changed and committed it in between your two reads.

Phantom read

You rund a query like SELECT ... WHERE salary > 50000 and get 12 rows. Later in the same transaction you run it again and get 13 rows - a new row appeared because another transaction inserted.

Lost update

Two transactions both read the same value, calculate a new value based on it and both write back. The second write silently overwrites the first. One update is simply lost.

A plain-English example: the dirty read

Transaction A deducts $200 from a bank account but has not committed. Transaction B reads the balance and sees the reduced amount. Transaction A then rolls back - the deduction never happened. But Transaction B already made a decision based on a balance that was never real. That is a dirty read.
