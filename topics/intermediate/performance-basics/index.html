Performance Basics

Before this topic: you should understand what transactions are (BEGIN / COMMIT blocks), how locks work (shared vs. exclusive, granularity) and what isolation levels control. This topic shows how all three interact to determine how fast — or how slow — your queries rund under concurrent load.

The previous topics in this chapter covered correctness: how transitions, locks and isolation levels protect your data from concurrent access problems. This topic is about cost — understanding why correct behavior sometimes comes with a performance price and what knobs you can actually turn.

Mental model: the database is a busy warehouse

Imagine a large warehouse during a flash sale. Thousands of workers (queries) arrive at the same time, each needing to find items on shelves (rows), note what they see and in some cases update the stock count. Three things determine how quickly each worker gets in and out:

1. How it navigates — index vs. scan

A worker with a precise shelf address goes straight there: one aisle, one shelf, done. That is an index seek — SQL Server jumps directly to the rows that match your WHERE clause. A worker without an address walks every single aisle checking every shelf: that is an index scan (or table scan). Scans read far more pages, take longer and hold any locks they acquire across every page they touch.

2. How long it holds its claim — transaction scope

Locks are held for the duration of the transaction. A worker who claims a shelf as soon as they arrive and doesn't release it until they've finished writing their paperwork, eaten lunch and made a phone call is blocking every other worker who needs that shelf. Transaction scope — what you put inside your BEGIN / COMMIT block — directly controls how long your locks are held. Wide scope (validation logic, calculations, logging all inside the transaction) means long lock hold times. Narrow scope (only the essential write inside the transaction) means short lock hold times.

3. What level of caution it operates under — isolation level

A worker operating under strict rules takes a formal claim on every shelf they glance at and holds it until their entire job is done (Serializable). A worker under relaxed rules glances at a shelf, notes what they see and releases the claim immediately (Read Committed). Higher isolation levels acquire more locks, hold them longer and therefore block more concurrent work. The trade-off is data consistency vs. throughput.

The four performance problems — and the levers that fix them

Problem Symptom Cause Lever
Slow reads Product pages time out under peak load Table scan instead on the lookup column Add a covering index on the lookup column
Read/write contention Inventory check blocks order insert Read holding shared locks unter Repeatable Read or Serializable Lower isolation level; consider READ_COMMITTED_SNAPSHOT
Long-blocking transaction Checkout sluggish site-wide during flash sale Transaction scope wraps validation, calculations and the write together Move everything except the essential write outside the transaction
Lock escalation Sudden site-wide slowdown at peak: a table lock appears Row-level locks escalate to a table lock when a treshold is crossed (~ 5,000 locks per object) Batch large writes; reduce rows touched per transaction

Measuring before you fix

Never guess at a performance problem. SQL Server provides diagnostic tools that show you exactly what is happening. Use these first, then decide which lever to pull.

SET STATISTICS TIME ON / SET STATISTICS IO ON

Run these before your query in SSMS. The output in the Messages tab tells you elapsed time (wall clock), CPU time and — most usefully — logical reads: how many 8KB data pages SQL Server read from the buffer cache to satisfy the query. A table scan on a large table can produc thousands of logical reads; an index seek on the same query might produce fewer than ten. Logical reads are the most reliable indicator of query efficiency because they are not affected by server load or caching effects the way elapsed time is.

Execution plans (Ctrl+M in SSMS)

The graphical execution plan shows you exactly which operations SQL Server chose to execute your query. Look for Index Seek (efficient — going straight to the right rows) vs. Table Scan or Scan (reading all rows). The thickness of the arrows between operations represents the estimated number of rows flowing through — a very thick arrow into an early Filter operation is often a sign that filtering happening too late.

sys.dm_exec_requests

When the database feels slow right now, this dynamic management view shows all currently running requests. The blocking_session_id column is the one to check first: a non-zero value means that session is waiting for a lock held by another session. The wait_type column tells you what kind of resource it is waiting for. See the Reference tab for the full query.

The practical order of attack: add the right index first (it often eliminates contention by reducing scan scope and lock footprint), then narrow transaction scope (reduces hold duration), the revisit isolation level only if contention persists. Changing isolation level without fixing indexes first is treating the symptom, not the cause.

Measurement tools

SET STATISTICS TIME ON / SET STATISTICS IO ON

Run before your query in SSMS. Output appears in the Messages tab.

SET STATISTICS TIME ON; SET STATISTICS IO ON; SELECT ProductId, StockQty FROM Products WHERE ProductId = 1042; SET STATISTICS TIME OFF; SET STATISTICS IO OFF;
Output field What it means What to look for
logical reads Pages read from the buffer cache Most reliabel efficiency indicator — not skewed by load or caching
physical reads Pages read from disk (not cached) High value means the data wasn't in memory — first-run effect is common
CPU time Processor time consumed High relative to elapsed time = CPU-bound; low = waiting on I/0 or locks
elapsed time Wall-clock duration Can be misleading on a busy server; use logical reads for comparison

sys.dm_exec_requests — live blocking check

Run while the database is under load to see what is blocking what.

SELECT r.session_id, r.blocking_session_id, -- non-zero = this session is blocked r.wait_type, -- e.g. LCK_M_S = waiting for shared lock r.wait_time, -- milliseconds spent waiting so far r.status, -- running, suspended, sleeping t.text AS query_text FROM sys.dm_exec_requests r CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t WHERE r.blocking_session_id > 0 -- only blocked sessions ORDER BY r.wait_time DESC;

sys.dm_tran_locks — lock Inventory

Shows all locks currently held or awaited. Useful when dm_exec_requests shows blocking but you need to understand what resource is contested. See the Locks topic for the full query.

Inside the transaction — keep short Outside the transaction — do before or after
The INSERT, UPDATE or DELETE that must be atomic Validation queries (SELECT to check stock, price, etc.)
Any reads that must be consistent with the write Calculations and business logic

Key terms

Term Definition
Logical read A page read from the SQL Server buffer cache (memory). The primary measure of query I/O cost.
Physical read A page read from disk because it was not in the buffer cache. Much slower than a logical read.
Index seek SQL Server navigates the index B-tree directly to matching rows. Efficient — reads a small number of pages.
Index scan / table scan SQL Server reads every row in the index or table to find matches. Inefficient at scale — reads every page.
Lock contention Two or more sessions competing for the same lock. One must wait (blocking) or one is killed (deadlock).
Lock escalation SQL Server converts many fine-grained row locks into a single coarser table lock to reduce memory overhead. Triggered at approximately 5,000 locks per object.
Transaction scope The set of operations enclosed in a BEGIN TRANSACTION / COMMIT block. locks are held for this entire duration.
blocking_session_id Column in sys.dm_exec_requests.A non-zero value means this session is waiting for a lock held by the identified session.

T-SQL dialect note

READ_COMMITTED_SNAPSHOT (RCSI) is a SQL Server / Azure SQL feature that changes Read Committed to use row versioning instead of shared locks, eliminating most reader/writer blocking without changing application isolation level settings. Enable at the database level:

ALTER DATABASE YourDatabase SET READ_COMMITTED_SNAPSHOT ON;

MySQL and PostgreSQL use MVCC (multi-version concurrency control) by default for all reads, so reader/writer blocking is less common in those engines without explicit configuration.
